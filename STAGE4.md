# 阶段 4：自动增量索引实现说明

## ✅ 已实现的功能

### 1. 文件监听

**实现位置：** `IndexScheduler`

**监听事件：**
- `create` - 文件创建
- `modify` - 文件修改
- `delete` - 文件删除
- `rename` - 文件重命名（视为删除旧文件 + 创建新文件）

**过滤规则：**
- 只处理 `.md` 文件
- 忽略非 Markdown 文件

### 2. Debounce 队列

**防抖延迟：** 2秒
- 文件变化后等待 2 秒再处理
- 如果 2 秒内有新变化，重置计时器
- 避免频繁编辑时重复索引

**队列机制：**
- 使用 `Map<path, update>` 存储待处理更新
- 同一文件的多次更新会被合并（只保留最新的）
- 例如：连续修改同一文件 10 次，只会处理 1 次

### 3. 并发控制

**批处理：** 每批 5 个文件
- 避免同时处理大量文件导致 API 限流
- 使用 `Promise.all` 并行处理每批文件
- 批次之间串行执行

**处理锁：**
- `isProcessing` 标志防止重复处理
- 队列处理中时，新的更新会继续累积
- 处理完成后自动处理累积的更新

### 4. 增量更新逻辑

**更新文件（create/modify）：**
```
1. 读取文件内容
2. 提取元数据（标题、标签、链接）
3. 分块（使用当前配置）
4. 删除旧的 chunks 和 embeddings
5. 生成新的 embeddings
6. 更新 chunks.json、embeddings.json、files.json
```

**删除文件（delete）：**
```
1. 找到该文件的所有 chunk IDs
2. 从 chunks.json 中删除
3. 从 embeddings.json 中删除
4. 从 files.json 中删除
```

**重命名文件（rename）：**
```
1. 删除旧路径的索引
2. 为新路径创建索引
```

### 5. 配置变化检测

**触发重建的配置：**
- `embeddingModel` - 向量模型变化
- `chunkSize` - 分块大小变化
- `overlap` - 重叠大小变化

**提示机制：**
- 修改这些配置时，显示警告通知
- 提示用户运行 "Build AI Index" 重建
- 不会自动重建（避免意外的大量 API 调用）

### 6. 启动和停止

**自动启动：**
- 插件加载时，如果 `autoIndexOnFileChange = true`，自动启动监听
- 设置页切换开关时，立即生效

**手动控制：**
- 设置页可以开关自动索引
- 关闭时停止文件监听，清空队列

### 7. 调试命令

**Show Index Queue Status：**
- 显示自动索引是否启用
- 显示当前队列大小

**Flush Index Queue：**
- 立即处理队列中的所有更新
- 不等待防抖延迟

## 🔄 工作流程

```
文件变化
  ↓
事件触发 (create/modify/delete/rename)
  ↓
加入队列 (Map<path, update>)
  ↓
防抖等待 (2秒)
  ↓
批量处理 (每批5个)
  ↓
增量更新索引
  ↓
完成
```

## 📊 性能优化

| 优化点 | 实现方式 | 效果 |
|--------|---------|------|
| 防抖 | 2秒延迟 | 避免频繁索引 |
| 合并更新 | Map 去重 | 同一文件只处理一次 |
| 批处理 | 每批5个 | 控制并发 |
| 增量更新 | 只更新变化的文件 | 避免全量重建 |

## ⚠️ 注意事项

### 不会自动重建的情况
- 修改 embedding model
- 修改 chunk size
- 修改 overlap

这些情况需要手动运行 "Build AI Index"。

### 队列持久化
- 队列是内存中的，重启插件会丢失
- 这是设计如此，避免启动时处理大量积压更新

### API 调用
- 每个文件更新都会调用 embedding API
- 建议使用支持批量的 API 提供商
- 注意 API 额度和限流
